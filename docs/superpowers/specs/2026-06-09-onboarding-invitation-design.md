# Onboarding and Invitation System Design

## 1. Overview
This document specifies the technical design for the first-time setup (onboarding) and the invitation code system for new users in OmniDrive. The goal is to create a seamless initial setup experience that automatically promotes the first user to a Super Admin, and subsequently restricts new signups by requiring valid invitation codes generated by the Super Admin.

## 2. Architecture & Database Changes
### 2.1. `users` Table Modifications
Add a new column to identify the super admin:
- `is_super_admin` (`INTEGER NOT NULL DEFAULT 0`)

### 2.2. New Table: `invitation_codes`
Create a new table to store and track invitation codes:
- `id` (`TEXT PRIMARY KEY`)
- `code` (`TEXT UNIQUE NOT NULL`)
- `created_by` (`TEXT NOT NULL REFERENCES users(id)`)
- `max_uses` (`INTEGER NOT NULL DEFAULT 1`) - 0 means unlimited.
- `used_count` (`INTEGER NOT NULL DEFAULT 0`)
- `expires_at` (`TEXT`) - Optional expiration date.
- `created_at` (`TEXT NOT NULL DEFAULT (datetime('now'))`)

## 3. Backend API Updates
### 3.1. Setup Status Endpoint
- **Route**: `GET /api/auth/setup-status`
- **Behavior**: Checks the total count of rows in the `users` table.
- **Response**: `{"isSetup": true}` if users exist, `{"isSetup": false}` if 0 users.

### 3.2. Registration Flow Modifications
- **Route**: `POST /api/auth/register` (and Google OAuth callback if applicable for new users)
- **Behavior**:
  - If `isSetup === false` (0 users in DB): Bypass invitation code check. Create the user and set `is_super_admin = 1`.
  - If `isSetup === true`: Require `invitation_code` in the payload. Validate the code against the `invitation_codes` table (`used_count < max_uses` and not expired). Increment `used_count` upon successful registration.

### 3.3. Admin Invitation Management
- **Routes**:
  - `GET /api/admin/invitations` (List all codes)
  - `POST /api/admin/invitations` (Create a new code)
  - `DELETE /api/admin/invitations/:id` (Revoke a code)
- **Middleware**: Must be protected by an `isAdmin` middleware that checks if `req.user.is_super_admin === 1`.

## 4. Frontend Flow & Components
### 4.1. Global Initialization Check
- On app load, the global state (Zustand) fetches `/api/auth/setup-status`.
- If `isSetup` is `false`, force-redirect the user to the `/setup` route.

### 4.2. Onboarding Screen (`/setup`)
- A dedicated, premium-looking screen displaying "Welcome to OmniDrive".
- Contains a form to create the first Super Admin account.

### 4.3. Registration Screen (`/register`)
- Updated to include a mandatory "Invitation Code" input field.
- Displays appropriate error messages if the code is invalid, expired, or fully used.

### 4.4. Admin Dashboard
- A new section under Settings or a dedicated Admin panel.
- Allows the Super Admin to generate new invitation codes, set usage limits, and view metrics (how many times a code has been used).

## 5. Security & Constraints
- The setup endpoint must accurately lock down once the first user is created to prevent unauthorized super admin creation.
- Invitation codes should be cryptographically secure or sufficiently random to prevent guessing.
- Route protection is crucial: only the Super Admin can access the `/api/admin/*` endpoints.
